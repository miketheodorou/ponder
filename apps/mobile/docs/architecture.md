## Key architectural choices

- **Tokens are values, presets come later.** `tokens.ts` is the verbatim spec from the design bundle, plus only what we've actually consumed: `serifDisplay: 64` (Welcome wordmark), `body2xl: 17` (form input text), `bodyLoose: 0.56` (tagline letter-spacing), `tightDisplay: -1.28` (wordmark letter-spacing). When component patterns repeat we'll add semantic presets (e.g. `theme.text.heroSerif`) — not yet.
- **Logical font names → concrete loaded names.** RN can't pick weights from one family, so `resolveFont({family:'sans', weight:'300', italic:true})` returns `'DMSans_300Light_Italic'`. Lora has no 300 weight; that case falls back to 400 inside the resolver instead of leaking into callers.
- **One source of truth for which fonts get loaded.** `fontsToLoad` from `theme/typography.ts` is what the root layout passes to `useFonts` — adding a new weight is a one-line change there.
- **ThemeProvider follows the OS scheme by default.** Both `ColorsDark` and `ColorsLight` are wired; switching is `<ThemeProvider scheme="light">` if we ever need it.
- **Three-layer background plumbing, all driven by tokens.** RN doesn't have an `<html>`-equivalent global background — there are three surfaces, and any of them being unset shows white seams during Stack push transitions:
  1. **Native window** — set via `expo-system-ui` (`SystemUI.setBackgroundColorAsync` in the root layout, re-runs when scheme changes).
  2. **React Navigation container theme** — `toNavigationTheme(theme)` adapts our Theme into the `@react-navigation/native` Theme shape (background, card, text, border, primary). Wrapped at the root with `<NavigationThemeProvider>`.
  3. **Stack `contentStyle.backgroundColor`** — set on every Stack (root and `(auth)`) so each screen's surface is themed even before the screen mounts its own `View`.
- **Top-anchor forms; don't fight the keyboard.** `KeyboardAvoidingView` on a vertically-centered form reflows the centered content as the keyboard rises — and that animation races the screen-push transition, which reads as jank. Forms are anchored from the top with explicit `paddingTop`; the keyboard slides up over the empty bottom area without moving anything. `autoFocus` on inputs works smoothly because the layout doesn't shift. If a form ever overflows on small devices, the fix is `ScrollView`, not KAV.
- **Safe-area handled at the screen.** The prototype's `paddingTop: 56` was a status-bar mock inside the device frame — on a real device that's `useSafeAreaInsets().top`. Bottom uses `Math.max(insets.bottom, theme.spacing.xxl)` to respect the home-indicator inset.
- **Icons are SVG, centralized in `components/icons.tsx`.** All glyphs share an `IconProps` shape (`size`, `color`, `strokeWidth`) and use the same stroke styling so weights stay consistent across screens. Replaces the rotated-border chevron approach used in `code.tsx`; that screen still uses the local one and can be migrated when convenient.
- **Auth routing via `Stack.Protected`, no root redirect file.** The root `app/_layout.tsx` wraps the Stack in two `Stack.Protected` blocks gated on `isLoggedIn` — when logged in, only `(app)` screens are in the route tree; when logged out, only `(auth)`. There is no `app/index.tsx` redirect — it would conflict with `(app)/index.tsx` (home) for URL `/`. `isLoggedIn` is currently hardcoded to `true` while we iterate on app screens; real auth will replace it. Flipping it to `false` to test the auth flow will leave `/` unrouted until an `(auth)` index is added (rename `welcome.tsx` to `index.tsx` or add a Redirect).
- **Catalogue is a router modal with a nested stack, not a custom sheet.** The catalogue is declared in `(app)/_layout.tsx` as `<Stack.Screen name='catalogue' options={{ presentation: 'modal', gestureEnabled: false }} />`. Tapping the Catalogue control on home calls `router.push('/catalogue')`, which mounts `(app)/catalogue/_layout.tsx` — a nested `Stack` — as the modal's content. Routes inside the modal (`/catalogue/quote/[id]`, `/catalogue/entry/[id]`, `/catalogue/entry/new`) push and pop within that nested stack while the modal stays up; `router.back()` from the root of the nested stack dismisses the entire modal. The earlier hand-rolled `CatalogueSheet` (Reanimated slide + Gesture.Pan handle + three absolute-positioned panes for list/detail/journal) is gone — the nested router stack handles cross-pane animation natively.
- **`presentation: 'modal'`, not `formSheet` or `pageSheet`.** `formSheet` + a nested `Stack` renders empty in expo-router 6 / react-native-screens 4 (the modal slides up but its content area is blank). `pageSheet` works but is still a sheet style we don't want. Settled on plain `modal`. To replace the missing native grabber, `(app)/catalogue/_layout.tsx` renders a `ChevronDown` Pressable above the nested Stack that calls `router.dismiss()` on tap — affordance + explicit close target in one. `gestureEnabled: false` kills iOS's interactive-dismiss so scrolling inside the catalogue list can't accidentally pull the modal down.

## Screen notes

- **(app)/index.tsx** is the post-auth landing screen (URL `/`). Header is a `Ponder` eyebrow on the left and a 32×32 plus button on the right (opens `CaptureModal`). The hero quote is rendered in Lora-400 at `serif4xl` (28pt), with `tightSerif` letter-spacing and a `marginTop: -32` optical lift so it reads slightly above true center. Footer stack: author·book eyebrow → full-width 0.5px hairline (uses `StyleSheet.hairlineWidth` so it stays crisp on every density) → centered `ChevronUp` + `Catalogue` eyebrow (tap to `router.push('/catalogue')`). Mounts `CaptureModal` and `Toast` as siblings; `captureOpen`/`toast` state lives on the screen. Uses `Math.max(insets.bottom, spacing.giant)` for the bottom inset to clear the home indicator. Hero quote is currently hardcoded to `q1`.
- **(app)/catalogue/index.tsx** is the catalogue list, the root of the modal's nested stack. Renders `CatalogueList` against seeded `QUOTES`. `onSelectQuote` pushes `/catalogue/quote/${id}` onto the nested stack. No top padding of its own — the catalogue layout's `paddingTop: insets.top` + dismiss-button row + `CatalogueList`'s internal header padding already handle the top space.
- **(app)/catalogue/quote/[id].tsx** renders `QuoteDetail` for the quote with the route `id`. `onBack` pops the nested stack, `onOpenEntry` pushes `/catalogue/entry/${entryId}`, `onNewEntry` pushes `/catalogue/entry/new?quoteId=${id}`.
- **(app)/catalogue/entry/[id].tsx** renders `JournalEntry` in read mode for a seeded entry from `ENTRIES`, resolves the linked quote from `QUOTES`. Back chevron pops to quote detail.
- **(app)/catalogue/entry/new.tsx** renders `JournalEntry` in composer mode by synthesizing an entry with `id: 'new'` and the `quoteId` search param. `onSave` is a no-op that just calls `router.back()` — persistence comes with the data layer (TanStack Query + AsyncStorage); the UI is presentational for now.
- **welcome.tsx** maps directly to the prototype: 32px-tall blank eyebrow row at top (kept for symmetry with Email/Code), Lora-400 64pt wordmark with -1.28 letter-spacing, italic DM Sans Light tagline below, "Sign in with email" PrimaryButton + faint terms eyebrow at the bottom. Button pushes to `/email`.
- **email.tsx** is top-anchored (no `flex: 1 + justifyContent: 'center'`): brand "Ponder" eyebrow row, then `paddingTop: 56` and the heading/subtitle/Field/Button stack. `autoFocus` on the Field opens the keyboard immediately. Submit pushes to `/code` with the email passed as a route param.
- **code.tsx** mirrors email.tsx's layout (header + top-anchored form with `paddingTop: 56`), with the back button on the left of the header instead of an empty slot. Subtitle reads the email from `useLocalSearchParams` and renders it inline in `textPrimary` against the muted body. Verify is a no-op (no `/home` yet); auto-submits via `CodeInput`'s `onComplete` once the user enters the sixth digit, since the iOS number-pad has no return key.

## Component notes

- **Field** wraps the Eyebrow-label + bottom-hairline TextInput pattern used by both auth and the future capture flow. It `forwardRef`s the underlying `TextInput` so screens can focus programmatically. **Italic placeholder workaround:** RN can't style `placeholderTextColor` independently from the input text style, so Field swaps the input's `fontFamily` to the italic variant (e.g. `DMSans_300Light_Italic`) while the value is empty, and back to the upright variant once the user types. The placeholder inherits the input's font, so it renders italic; the user's text doesn't.
- **CodeInput** renders N visual slot Views with a single transparent `TextInput` absolutely positioned over them (`opacity: 0`, `caretHidden`, `selectionColor: 'transparent'`). Native taps focus the input; the OS keyboard handles all input mechanics. Non-digits are filtered inside the component, so consumers always get a clean numeric string. Active slot has a blinking caret driven by a `setInterval` (gated on focus). **Auto-fill:** `textContentType="oneTimeCode"` (iOS) and `autoComplete="sms-otp"` (Android) let the OS suggest codes received by SMS.
- **Inline chevron in code.tsx** is drawn with a rotated bordered square (`borderTopWidth + borderLeftWidth + transform: rotate(-45deg)`) rather than pulling in `react-native-svg` for a single glyph. When more icons land, replace with a real icon component — don't pile up rotated-border chevrons across the codebase.
- **catalogue/\_layout.tsx renders the dismiss control above the nested Stack.** Because `gestureEnabled: false` on the parent stack disables iOS interactive-dismiss, the layout adds a top `ChevronDown` Pressable (`router.dismiss()` on tap) — affordance + only available close target from the catalogue list. The layout also owns `paddingTop: insets.top` so the dismiss button clears the status bar on platforms where the modal frame doesn't (Android full-screen modal). Inner screens (`/catalogue/quote/[id]`, `/catalogue/entry/[id]`, `/catalogue/entry/new`) use `NavHeader` for in-stack back navigation.
- **CatalogueList** owns its own `query`/`activeChip` state. Tag filter logic mirrors the prototype: `all` and `recent` ignore the tag filter; everything else filters `quote.tags.includes(chip.id)`. The `recent` chip additionally sorts by parsed date descending (the mock data is already in that order, but the sort makes it correct as data changes). Search matches case-insensitively against text/author/book. Filter chips are a horizontal `ScrollView` (8 items — virtualization unnecessary). Quote rows are a vertical `ScrollView` with hairline dividers between rows; the first row has no top border. `keyboardShouldPersistTaps="handled"` lets users tap a row while the search keyboard is open without it dismissing first. `onSelectQuote` is wired by the route (`catalogue/index.tsx`) to push `/catalogue/quote/${id}`.
- **QuoteDetail** is the read view of one quote with two inline edit affordances:
  - **+ tag** — the dashed pill swaps to a `TextInput` on tap. Submit (return key or blur) appends the trimmed lowercased value to the local `tags` array if it isn't already there. Empty submits collapse the input back to the placeholder.
  - **Notes** — tap the notes block (or the "Add notes…" placeholder when empty) to swap into a multiline `TextInput`. Blur swaps back to display. Display is italic serif; edit is upright serif so the caret + selection don't compete with italic letterforms.
  Both edits are **ephemeral** — held in QuoteDetail's local state and lost when the component remounts (which happens when the sheet closes and reopens). This is fine for a design pass; persistence comes with the data layer. Journal section: linked entry rows push journal-pane on tap; "+ New entry" pushes a synthesized entry with `id: 'new'`.
- **JournalEntry** is a single screen with two modes — read view of an existing entry (`Text`) and a composer for a new entry (`TextInput multiline autoFocus`). The "new" marker is the prototype's pattern: a synthesized entry with `id: 'new'` carrying the linking `quoteId`, an empty body, and a "New entry" date label. New mode renders a Save action in the NavHeader's right slot — disabled until the body has non-whitespace content. Tapping Save sends the trimmed body to `onSave`. Currently wired by `catalogue/entry/new.tsx` to a no-op `router.back()` — persistence is deferred until the real data layer. The linked quote is rendered as an italic serif preview block with a left hairline border, capped at 3 lines.
- **NavHeader** is the shared back-chevron + label header used inside the catalogue stack screens. Sits outside the scrolling content so the back affordance stays put as the user scrolls. Accepts an optional `right` slot — currently used for the Save action on JournalEntry; the slot is general-purpose so future screens (e.g. capture confirmation) can drop in their own action without bespoke headers.
- **CaptureModal** is the 4-step capture wizard. Step 0 (camera) renders a mock viewfinder with frame corners + faux book-page bars and a 78pt iOS-style shutter (outer hairline ring, inner cream fill). The shutter just shoves `OCR_SAMPLE` into `text` and advances. Step 1 (edit) is a serif `TextInput multiline autoFocus` for trimming the OCR text down to the quote. Step 2 (context) is three label-left input-right rows — Book / Author / Page (page uses `keyboardType="number-pad"`). Step 3 (confirm) renders a `bgRaised` card with the trimmed quote + Book/Author/Page rows and a Save button. The top bar swaps a Close icon for a back chevron once `step > 0`, and the four progress dots Reanimate width 5 → 16 between active/inactive (color flips instantly — color animations on a 5pt pill aren't worth interpolateColor's overhead). Steps 1–3 share a `CaptureFooter` (hairline divider + `PrimaryButton`) that's disabled until the relevant fields validate.
- **Toast** is a small animated pill mounted at the bottom of the home screen. The component just animates between visible/hidden based on `message`; the auto-dismiss timer is HomeScreen's job — saves push a string into `toast` state and clear it via `setTimeout(2200ms)`. Consecutive toasts replace each other cleanly because the timer is held in a ref and cleared on each new push.
- **No session state — UI is presentational against seeded data.** Earlier the home screen owned `sessionQuotes` / `sessionEntries` stores fed by `CaptureModal` and `CatalogueSheet`. Those were removed: the catalogue routes read directly from seeded `QUOTES`/`ENTRIES`, `CaptureModal`'s Save is a no-op (closes + toasts), and `JournalEntry`'s Save just pops the stack. Real persistence will land with TanStack Query + an AsyncStorage offline layer; until then, nothing about the UI should pretend to write through.
- **Deliberate inconsistency in edit persistence (still ephemeral).** Tag and notes edits on QuoteDetail are local to that component and lost on remount. Save buttons in JournalEntry's composer and CaptureModal are wired but currently no-op (they don't write to any store, since none exists). When the data layer is real, all three should funnel through the same mutation path.
- **Capture flow is a slide-up modal, mounted on HomeScreen.** Absolute-positioned `Animated.View` that animates `translateY` between `0` (open) and `windowHeight` (closed) over 380ms. Stays mounted for the home screen's lifetime, so the `book`/`author` fields persist across captures (the "pre-filled from your last capture" hint). Only `text` and `page` reset on each open — they're specific to the new capture; book/author are commonly the same source. **Note:** unlike the catalogue, capture is *not* a router modal — it pre-dates the catalogue's migration. Could be unified with the same `presentation: 'modal'` pattern when convenient, but the existing custom slide already works.
- **OCR is mocked.** `OCR_SAMPLE` is a const string the camera step shoves into `text` when the shutter is pressed. The viewfinder itself is a visual mock — frame corners + a faux book page made of width-varied bars — so we can iterate on UX without `expo-camera`, permissions, or a vision API. Replace by wiring `expo-camera` to capture a photo URI and a vision API call to produce the text — the rest of the flow doesn't change.
- **KeyboardAvoidingView in capture.** The capture modal is the one place where we use `KeyboardAvoidingView` (the architecture's general advice is to avoid it). Justification: header/body/footer layout (not a centered form); the body shrinks under the keyboard rather than reflowing centered content, so the jank-with-screen-push race the auth screens warned about doesn't apply. Without it, the bottom CTA gets covered by the keyboard during the multi-line OCR edit and there's no return key to dismiss.
