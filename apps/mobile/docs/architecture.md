## Key architectural choices

- **API client is a ky singleton with a registered Clerk token getter.** `api/client.ts` exports a ky instance created with `baseUrl: API_URL`. Because Clerk's `getToken()` only exists inside a React hook but ky is module-level, `registerAuthTokenGetter(getter)` stores a module-level reference that the `beforeRequest` hook calls on every request. The root layout binds it via a `useEffect` on `getToken` inside `ThemedShell` (already inside `ClerkProvider`). This means every fetch automatically carries `Authorization: Bearer <token>` — no per-call auth plumbing in query functions.
- **API URL is derived from Metro's host in development.** `lib/api.ts` reads `Constants.expoConfig.hostUri` (set by Metro's bundler) and derives the API URL as `http://${host}:3000`. This works transparently for iOS Simulator (localhost), Android Emulator, and physical devices on Expo Go. In production (`!__DEV__`) it falls back to `EXPO_PUBLIC_API_URL`. No per-platform environment vars needed.
- **Types come from `@ponder/db/schema`, composed at the boundary.** API client functions import `Quote` and `JournalEntry` from `@ponder/db/schema` (workspace dep, `import type` only — zero runtime impact on the bundle). A small `WireQuote`/`WireJournalEntry` helper narrows `id` and `createdAt` to required at each api function's return type (they're optional on `$inferInsert` but always present in SELECT responses). Components receive the schema types and compose extra fields (`themes: string[]`, `journalEntries: JournalEntryPreview[]`) inline — no parallel type declarations in the mobile package.
- **TanStack Query is the data layer.** `lib/query-client.ts` exports a singleton `QueryClient`; `QueryClientProvider` wraps the app tree just inside `ClerkProvider`. All data screens use `useQuery` — query keys follow the pattern `['quotes']`, `['quotes', id]`, `['quotes', 'today']`, `['journal-entries', id]`. Cache is shared across navigations (e.g. navigating quote detail → journal entry → back hits the `['quotes', id]` cache already warmed by detail). Mutations (create journal entry, save quote) are not yet wired — Save actions still call `router.back()`.
- **Tokens are values, presets come later.** `tokens.ts` is the verbatim spec from the design bundle, plus only what we've actually consumed: `serifDisplay: 64` (Welcome wordmark), `body2xl: 17` (form input text), `bodyLoose: 0.56` (tagline letter-spacing), `tightDisplay: -1.28` (wordmark letter-spacing). When component patterns repeat we'll add semantic presets (e.g. `theme.text.heroSerif`) — not yet.
- **Logical font names → concrete loaded names.** RN can't pick weights from one family, so `resolveFont({family:'sans', weight:'300', italic:true})` returns `'DMSans_300Light_Italic'`. Lora has no 300 weight; that case falls back to 400 inside the resolver instead of leaking into callers.
- **One source of truth for which fonts get loaded.** `fontsToLoad` from `theme/typography.ts` is what the root layout passes to `useFonts` — adding a new weight is a one-line change there.
- **ThemeProvider owns appearance state, persisted via AsyncStorage.** `appearance` is `'system' | 'light' | 'dark'` (key `ponder.appearance`). `useTheme()` returns the resolved Theme; `useAppearance()` returns `{ appearance, setAppearance, ready }`. `setAppearance` writes back through the same hook fire-and-forget; the in-memory value is the source of truth for the session and a failed write just falls back to `'system'` next cold boot. **`appearance` is the first persisted UI state in the app** — everything else (catalogue tags, journal composer, capture draft) is still ephemeral until the data layer lands.
- **Boot gate is a single AND of three signals.** `ThemedShell` holds `SplashScreen` up until `(fontsLoaded || fontError) && isLoaded && appearanceReady`. ClerkProvider mounts immediately on RootLayout render so the session check, font load, and AsyncStorage read all happen in parallel rather than chained. Without the appearance gate, the first frame would render with the default scheme and then snap to the persisted choice; without the Clerk gate, the auth/app trees would mount before `isSignedIn` was known and produce a wrong-tree flash.
- **Three-layer background plumbing, all driven by tokens.** RN doesn't have an `<html>`-equivalent global background — there are three surfaces, and any of them being unset shows white seams during Stack push transitions:
  1. **Native window** — set via `expo-system-ui` (`SystemUI.setBackgroundColorAsync` in the root layout, re-runs when scheme changes).
  2. **React Navigation container theme** — `toNavigationTheme(theme)` adapts our Theme into the `@react-navigation/native` Theme shape (background, card, text, border, primary). Wrapped at the root with `<NavigationThemeProvider>`.
  3. **Stack `contentStyle.backgroundColor`** — set on every Stack (root and `(auth)`) so each screen's surface is themed even before the screen mounts its own `View`. The `settings` screen overrides this with `backgroundColor: 'transparent'` so the home route shows through its scrim.
- **Two keyboard patterns, by form shape.** Auth screens (welcome/email/code) anchor their content from the top with explicit `paddingTop: 56` — the keyboard slides up over empty bottom space without reflowing anything, which means `autoFocus` on the input doesn't race the screen-push transition (the layout never shifts). The router-modal flows (capture and catalogue) use `KeyboardAwareScrollView` from `react-native-keyboard-controller` for any screen with text input — it scrolls just enough to keep the focused input + `bottomOffset` clear of the keyboard. Each modal layout mounts a single `<KeyboardToolbar />` as a sibling to the nested Stack so the system-style Done / Previous / Next bar appears above the keyboard for every screen in the flow, cross-platform. Old `KeyboardAvoidingView` usage was removed — RN's built-in KAV was unreliable inside our native-stack modal screens. Don't reach for KAV by default.
- **`react-native-keyboard-controller` is wired at the root.** `<KeyboardProvider>` wraps everything inside `GestureHandlerRootView` in `app/_layout.tsx`. The `KeyboardToolbar` is mounted once per modal layout (`(app)/capture/_layout.tsx` and `(app)/catalogue/_layout.tsx`) so it's available across every screen in that flow without per-screen wiring. Auth screens still use stock RN since their top-anchored pattern doesn't need keyboard avoidance at all.
- **Safe-area handled at the screen.** The prototype's `paddingTop: 56` was a status-bar mock inside the device frame — on a real device that's `useSafeAreaInsets().top`. Bottom uses `Math.max(insets.bottom, theme.spacing.xxl)` to respect the home-indicator inset.
- **Icons are SVG, centralized in `components/icons.tsx`.** All glyphs share an `IconProps` shape (`size`, `color`, `strokeWidth`) and use the same stroke styling so weights stay consistent across screens. Replaces the rotated-border chevron approach used in `code.tsx`; that screen still uses the local one and can be migrated when convenient.
- **Auth is Clerk + `@clerk/expo` v3 signal API.** `ClerkProvider` lives at the root with `tokenCache` from `@clerk/expo/token-cache` (SecureStore-backed). The publishable key comes from `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY`. We use the new signal hooks (`useAuth`, `useSignIn`, `useSignUp`, `useUser`, `useClerk`) — **not** the legacy resource API at `@clerk/expo/legacy`. Methods on signal-API resources return `{ error: ClerkError | null }` instead of throwing, so the auth screens branch on the returned `error` rather than `try/catch`. Plugins `expo-secure-store` and `@clerk/expo` are registered in `app.json` for prebuild/EAS; in dev with Expo Go this code doesn't run (Clerk requires a dev client).
- **Unified email-OTP flow via `signUpIfMissing`.** `email.tsx` calls `signIn.create({ identifier, signUpIfMissing: true })` and **checks `signIn.isTransferable` BEFORE inspecting the returned error.** With `signUpIfMissing`, Clerk still surfaces an error (e.g. "Couldn't find your account") when the identifier is unknown, but it ALSO marks the resource transferable — so bailing on the error first would skip the transfer and dead-end legitimate new users. When `isTransferable`, we call `signUp.create({ transfer: true })` + `signUp.verifications.sendEmailCode()` and route to `/code?mode=signup`; otherwise `signIn.emailCode.sendCode()` + `/code?mode=signin`. `code.tsx` reads `mode` and dispatches `emailCode.verifyCode` vs `verifications.verifyEmailCode` accordingly. The `/code` subtitle splices " to create your account" into the body copy when `mode === 'signup'` so a typo'd existing user has a chance to catch it before completing verification.
- **`finalize({ navigate })` only handles session tasks; route swap is `Stack.Protected`'s job.** The web Clerk docs' navigate callback calls `router.push('/')` for the no-task case, but on Expo Router that races the `(auth)→(app)` guard flip and produces `"REPLACE not handled"` errors. The callback here is intentionally one branch: if `session.currentTask` is set, throw with the task key (we don't have screens for `'choose-organization' | 'reset-password' | 'setup-mfa'` yet); otherwise no-op and let the guard mount `(app)`. When task-producing features get enabled in the Clerk dashboard, add a dedicated route (e.g. `app/task/[key].tsx` protected by `isSignedIn && session?.currentTask`) and route to it from the callback.
- **Auth routing via `Stack.Protected`, driven by `useAuth().isSignedIn`.** The root `app/_layout.tsx` wraps the Stack in two `Stack.Protected` blocks — when `isSignedIn`, only `(app)` screens are in the route tree; when not, only `(auth)`. Sign-out (`useClerk().signOut()` from the settings drawer) flips the guard and unmounts `(app)` automatically; the settings screen unmounts as part of that swap. No manual navigation calls; no `app/index.tsx` redirect (it would conflict with `(app)/index.tsx`).
- **`SplashScreen.hideAsync()` is wrapped in `.catch`.** Expo-router also calls hideAsync internally on route changes, and iOS rejects with "No native splash screen registered for given view controller" when called against a non-root view controller (which it creates during `(app)→(auth)` swaps after sign-out). The splash is already hidden at that point; the catch suppresses the unhandled-rejection warning. This is a known expo-router / expo-splash-screen interaction, not a bug in our code.
- **Dev-only affordances are `__DEV__`-gated, not env-flagged.** `__DEV__` is the RN-native compile-time boolean — `true` in Metro dev builds, `false` in production. Metro dead-code-eliminates the entire guarded block when false, so the affordance, its callbacks, and any package imports referenced only inside it never ship. Today this gates one thing: the "Copy session token" eyebrow in the settings drawer. It calls `getToken({ template: 'dev' })`, where `dev` is a JWT template you configure once in the Clerk dashboard (Sessions → JWT Templates) with the longest expiry your plan allows — the default 60-second session token is too short for iterative Postman work against `apps/server`. The template name lives as a constant in `settings.tsx`, not env — it's a Clerk-dashboard convention, not a secret or a per-environment value. If the template isn't configured, the button falls back to `getToken()` (default short-lived token) and labels the action `"Copied (default)"` so the missing config is self-explanatory.
- **Catalogue is a router modal with a nested stack, not a custom sheet.** The catalogue is declared in `(app)/_layout.tsx` as `<Stack.Screen name='catalogue' options={{ presentation: 'modal', gestureEnabled: false }} />`. Tapping the Catalogue control on home calls `router.push('/catalogue')`, which mounts `(app)/catalogue/_layout.tsx` — a nested `Stack` — as the modal's content. Routes inside the modal (`/catalogue/quote/[id]`, `/catalogue/entry/[id]`, `/catalogue/entry/new`) push and pop within that nested stack while the modal stays up; `router.back()` from the root of the nested stack dismisses the entire modal. The earlier hand-rolled `CatalogueSheet` (Reanimated slide + Gesture.Pan handle + three absolute-positioned panes for list/detail/journal) is gone — the nested router stack handles cross-pane animation natively.
- **`presentation: 'modal'`, not `formSheet` or `pageSheet`.** `formSheet` + a nested `Stack` renders empty in expo-router 6 / react-native-screens 4 (the modal slides up but its content area is blank). `pageSheet` works but is still a sheet style we don't want. Settled on plain `modal`. To replace the missing native grabber, `(app)/catalogue/_layout.tsx` renders a `ChevronDown` Pressable above the nested Stack that calls `router.dismiss()` on tap — affordance + explicit close target in one. `gestureEnabled: false` kills iOS's interactive-dismiss so scrolling inside the catalogue list can't accidentally pull the modal down.
- **Settings is a `transparentModal`, not a `modal`.** iOS' native modal presentation slides up from the bottom; we want a left-slide drawer. `transparentModal` keeps the previous (home) route mounted underneath with no system entrance animation; the settings screen owns its own animation (Reanimated `progress` shared value drives both the panel `translateX` and the scrim opacity, 380ms / 320ms with the iOS sheet bezier `[0.32, 0.72, 0, 1]`). Drag-to-dismiss uses `Gesture.Pan` — leftward drag past 100px commits to dismiss; release short of that snaps back. The drop shadow opacity is animated alongside `progress` so the right-edge shadow doesn't cling to screen-left as the panel exits. **Logout is a real Clerk call**, not a stub: `useClerk().signOut()` runs, the auth state flips, the `Stack.Protected` guard at root swaps trees, and the settings route unmounts as part of that swap (no manual navigation).
- **Capture is a router modal mirroring catalogue, with a Context for wizard draft state.** Same `presentation: 'modal'` + `gestureEnabled: false` declaration in `(app)/_layout.tsx`. The plus button on home calls `router.push('/capture')`, which mounts `(app)/capture/_layout.tsx`. Unlike catalogue, capture is a four-step wizard with shared state across steps (the OCR-trimmed `text`, plus `book`/`author`/`page`) — that state lives in a `CaptureDraftProvider` (Context) wrapping the nested `Stack`, so it persists as the user pushes from camera → edit → context → confirm and resets when the modal closes. The layout owns the chrome: Close-or-Back affordance on the left (close at step 0, back at step 1+), "Capture" eyebrow centered, and four animated progress dots on the right (Reanimated `withTiming` on width 5 → 16). The current step is derived from `useSegments()` so the dots and back-vs-close logic stay in sync without screens propagating state upward.

## Screen notes

- **(app)/index.tsx** is the post-auth landing screen (URL `/`). Header is a `Ponder` eyebrow on the left (`Pressable` → `router.push('/settings')`) and a 32×32 plus button on the right (`router.push('/capture')`). The hero quote is rendered in Lora-400 at `serif4xl` (28pt), with `tightSerif` letter-spacing and a `marginTop: -32` optical lift so it reads slightly above true center. Footer stack: author·book eyebrow → full-width 0.5px hairline → centered `ChevronUp` + `Catalogue` eyebrow (`router.push('/catalogue')`). Hero quote is fetched via `useQuery(['quotes','today'], getTodaysQuote)` — returns the user's most recently created quote (`id`, `text`, `bookTitle`, `authorName`). The chrome (header/footer/hairline) mounts immediately; the quote text and attribution conditionally render once `data` arrives so there's no flash of blank content. Previously hardcoded to `q1`.
- **(app)/settings.tsx** is a left-slide drawer mounted as a `transparentModal`. Appearance picker is three cards in a row (System / Light / Dark) wired to `useAppearance`; the active card flips immediately and the new value persists to AsyncStorage in the background. Email is read from `useUser().primaryEmailAddress`. Log out button is `PrimaryButton variant='outline'` with a label-swap loading state (`"Signing out…"` while `submitting`) and an inline error mirrored from the auth screens' pattern — on success it doesn't navigate; `Stack.Protected` unmounts `(app)` when `isSignedIn` flips. A `__DEV__`-gated "Copy session token" eyebrow sits between the logout block and the version line — taps call `getToken({ template: 'dev' })`, copy the JWT to the clipboard via `expo-clipboard`, and flip the label to `"Copied"` for ~1.8s. The bottom corner shows a "Ponder · v1.0" eyebrow in `textFaint`.
- **(app)/capture/index.tsx** is the camera step (step 0). Renders a mock viewfinder (frame corners + faux book-page bars) and a 78pt iOS-style shutter (outer hairline ring, inner cream fill). Tapping the shutter writes `OCR_SAMPLE` into the draft via `useCaptureDraft().setText`, then `router.push('/capture/edit')`. No `CaptureFooter` on this step — the shutter is the action.
- **(app)/capture/edit.tsx** is the OCR-trim step (step 1). Multiline serif `TextInput autoFocus` for trimming the OCR sample down to the quote. Wrapped in `KeyboardAwareScrollView` (`bottomOffset={90}` to clear the CaptureFooter) so the footer Next button stays visible above the keyboard. `<KeyboardToolbar />` is mounted as a sibling — Done dismisses the keyboard, and the Prev/Next chevrons cycle focus across any other inputs on screen (none here, so they're inert). Next pushes `/capture/context` (disabled until the text trims to non-empty).
- **(app)/capture/context.tsx** is the context step (step 2). Three `CaptureField` rows (Book / Author / Page; Page uses `keyboardType="number-pad"`), wrapped in `KeyboardAwareScrollView` + `<KeyboardToolbar />`. The toolbar's Prev/Next chevrons let the user move across the three fields without tapping into each. Next pushes `/capture/confirm` (disabled until Book + Author are non-empty).
- **(app)/capture/confirm.tsx** is the review + save step (step 3). Renders a `bgRaised` card with the trimmed quote + Book/Author/Page rows. Save calls `router.replace('/')` — capture is only ever entered from home, so navigating to home is the simplest way to close the entire modal stack in one step. `router.dismiss()`/`dismissAll()` only operate on the inner nested capture stack (which is why `dismissAll` landed on `/capture` instead of dismissing the modal). Persistence comes with the data layer.
- **(app)/catalogue/index.tsx** is the catalogue list, the root of the modal's nested stack. Fetches the user's full quote list via `useQuery(['quotes'], getQuotes)` and passes `data ?? []` to `CatalogueList`. `onSelectQuote` pushes `/catalogue/quote/${id}`. No top padding of its own — catalogue layout + CatalogueList's internal header handle it.
- **(app)/catalogue/quote/[id].tsx** fetches via `useQuery(['quotes', id], () => getQuoteById(id))` (enabled only when `id` is set). Passes `data ?? null` to `QuoteDetail`. `onBack` pops the nested stack, `onOpenEntry` pushes `/catalogue/entry/${entryId}`, `onNewEntry` pushes `/catalogue/entry/new?quoteId=${id}`.
- **(app)/catalogue/entry/[id].tsx** fetches the entry via `useQuery(['journal-entries', id], () => getJournalEntryById(id))`, then chains a dependent `useQuery(['quotes', entry.quoteId], () => getQuoteById(entry.quoteId))` for the linked quote context block. The quote query shares its cache key with the detail screen, so navigating from detail to entry is a cache hit.
- **(app)/catalogue/entry/new.tsx** fetches the linked quote via `useQuery(['quotes', quoteId])` (shares cache with detail). Synthesizes `{ id: 'new', quoteId, userId: '', content: '' }` as the draft — `JournalEntry` reads `id === 'new'` to enter composer mode. `onSave` still calls `router.back()`; the create mutation isn't wired yet.
- **(auth)/index.tsx** (formerly `welcome.tsx`) maps directly to the prototype: 32px-tall blank eyebrow row at top (kept for symmetry with Email/Code), Lora-400 64pt wordmark with -1.28 letter-spacing, italic DM Sans Light tagline below, "Sign in with email" PrimaryButton + faint terms eyebrow at the bottom. Button pushes to `/email`. Lives at the group index (not `welcome.tsx`) so the default route inside `(auth)` is welcome without needing `initialRouteName` plumbing in the layout.
- **(auth)/email.tsx** is top-anchored: brand "Ponder" eyebrow row, then `paddingTop: 56` and the heading/subtitle/Field/Button stack. `autoFocus` on the Field opens the keyboard immediately. Submit drives the unified Clerk flow described above (`signIn.create({ identifier, signUpIfMissing: true })`, branch on `isTransferable`, then transfer or `emailCode.sendCode`) and routes to `/code` with `email` + `mode` params. Inline error appears below the field for non-transferable failures; the "user not found" case is silent (transfer happens behind the scenes).
- **(auth)/code.tsx** mirrors email.tsx's layout (header + top-anchored form with `paddingTop: 56`), with the back button on the left of the header instead of an empty slot. Subtitle reads the email from `useLocalSearchParams` and splices " to create your account" into the body when `mode === 'signup'`. Verify runs `signIn.emailCode.verifyCode` (signin) or `signUp.verifications.verifyEmailCode` (signup); on `status === 'complete'` it calls `finalize({ navigate: finalizeNavigate })`. Auto-submits via `CodeInput`'s `onComplete` once the user enters the sixth digit, since the iOS number-pad has no return key. Resend reuses the same method that sent the original (`signIn.emailCode.sendCode` or `signUp.verifications.sendEmailCode`).

## Component notes

- **Field** wraps the Eyebrow-label + bottom-hairline TextInput pattern used by both auth and the future capture flow. It `forwardRef`s the underlying `TextInput` so screens can focus programmatically. **Italic placeholder workaround:** RN can't style `placeholderTextColor` independently from the input text style, so Field swaps the input's `fontFamily` to the italic variant (e.g. `DMSans_300Light_Italic`) while the value is empty, and back to the upright variant once the user types. The placeholder inherits the input's font, so it renders italic; the user's text doesn't.
- **CodeInput** renders N visual slot Views with a single transparent `TextInput` absolutely positioned over them. The TextInput has `color: 'transparent'` (so its own text never paints over the slot grid) but **its opacity stays at 1 and `selectionColor` is unset** — those two together are what give iOS a visible anchor for its native edit menu, which is how tap-to-Paste works on a focused input. `caretHidden` is still on (our custom slot caret is the only visible cursor). Non-digits are filtered inside the component, so consumers always get a clean numeric string. Active slot has a blinking caret driven by a `setInterval` (gated on focus). **Auto-fill:** `textContentType="oneTimeCode"` (iOS) and `autoComplete="sms-otp"` (Android) let the OS suggest codes received by SMS in the QuickType bar above the keyboard.
